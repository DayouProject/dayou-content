# 每周工作流

这套流程的目标不是“写很多内容”，而是稳定把 2 到 3 个概念转成 3 到 7 条已发布、可复盘的视频。

## Mon-Sun 时间线

```text
周一    周二        周三          周四          周五          周六          周日
-----   ----------  ------------  ------------  ------------  ------------  ------------
知识库   提交概念     审脚本         补充修订       -             -             更新知识库备注
Lexa    接收输入     跑脚本         改脚本并定稿    生成视频       盯发布异常      复盘产线
宣发    开平台排期    准备标题/标签   确认发布清单    接收成片        发布并互动      提交周报
```

## 交接协议

### 1. 概念如何变成脚本

| 谁做 | 做什么 | 在哪做 | 什么时候做 | 什么算合格 |
|---|---|---|---|---|
| 知识库负责人 | 新建 1 个 Content Idea Issue，提交 2 到 3 个概念 | GitHub Issue | 周一 12:00 前 | 每个概念都有经典来源、现代角度、目标情绪、不要怎么讲 |
| Lexa | 检查输入是否可生产，不完整就退回补充 | GitHub Issue | 周一 18:00 前 | 明确回复“已接收”或“补齐后再生成” |
| Lexa | 运行管线生成首批脚本 | `pipeline.py` + `workspace/scripts/` | 周二 | 至少生成 3 条可审脚本，并在 Issue 贴路径 |
| 知识库负责人 | 审核脚本并逐条标记通过 / 需改 / 拒绝 | GitHub Issue + `workspace/scripts/` | 周三 18:00 前 | 指到具体句子，不写空话，不只说“感觉不对” |

### 2. 脚本如何变成视频

| 谁做 | 做什么 | 在哪做 | 什么时候做 | 什么算合格 |
|---|---|---|---|---|
| Lexa | 根据审核意见修订脚本并定稿 | `workspace/scripts/` + GitHub Issue | 周三晚前 | 待发布脚本版本明确，Issue 中能看到最终状态 |
| Lexa | 生成视频并输出到 `workspace/videos/` | HeyGen + 本仓库 | 周四 | 至少 3 条视频可播放、口型和字幕无明显错误 |
| 宣发负责人 | 抽检前 10 秒、字幕、封面文案、平台适配 | `workspace/videos/` + 平台后台 | 周四当日 | 能明确写“可发”或指出具体问题点 |

### 3. 视频如何变成已发布内容

| 谁做 | 做什么 | 在哪做 | 什么时候做 | 什么算合格 |
|---|---|---|---|---|
| 宣发负责人 | 手动或半自动发布首批视频 | TikTok、Instagram | 周五开始 | 每条视频都有发布时间与公开链接 |
| 宣发负责人 | Reddit 做轻量社区种草，不硬广 | Reddit | 周五到周日 | 每天至少 1 条相关互动，记录社区与话题 |
| 宣发负责人 | 记录基础数据和异常 | 平台后台 + GitHub Issue | 每天 | 链接、播放、互动、异常都有记录 |
| 宣发负责人 | 提交 Weekly Report Issue | GitHub Issue | 周日 | 报告能直接支持下周选题与排期 |

## 出问题时怎么处理

| 情况 | 第一责任人 | 立刻做什么 | 升级条件 | 关闭标准 |
|---|---|---|---|---|
| 脚本被知识库负责人拒绝 | 知识库负责人 | 在 Issue 写清哪一句不行、为什么不行、建议怎么改 | 当天内双方对改法仍不一致 | Lexa 提交修订版，知识库负责人明确回复“通过” |
| 视频质量差 | 宣发负责人 | 标出具体时间点，例如“00:07 口型错位”或“字幕术语不对” | 需要当天发但无法修好 | Lexa 重新生成，宣发负责人复检通过 |
| 内容被平台标记或限流 | 宣发负责人 | 先截图保留证据，暂停同批次扩散，发微信提醒 | 涉及账号风险、连续 2 条以上异常 | Issue 记录原因、处理动作、后续是否重发或放弃 |
| 自动发布失败 | Lexa | 检查 n8n、Webhook、账号授权 | 影响当天发布时间 | 改为手动发，并在 Issue 补充原因与修复结果 |

## 沟通协议

| 用什么 | 什么时候用 | 不要拿来做什么 | 补充要求 |
|---|---|---|---|
| GitHub Issue | 选题、脚本审核、交接、周报、异常留痕 | 不能等的紧急催办 | 所有正式决定都要留在这里 |
| 微信 | 今天必须处理的事；账号登不上；平台告警；临时改排期 | 长期任务管理；口头定稿 | 微信上说完，30 分钟内补回对应 Issue |
| 仓库路径 | 看脚本、看视频、对照版本 | 替代审批与评论 | 所有文件路径都要回贴到 Issue，不能只口头说 |

## 一周结束时要留下什么

```text
1. 1 个 Content Idea Issue
2. 1 轮脚本审核记录
3. 1 批视频路径与发布链接
4. 1 个 Weekly Report Issue
```

没有这 4 类记录，就不算真正跑完一周。
