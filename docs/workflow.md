# 每周工作流

这条流程的目标很简单：把传统智慧稳定转成一周 7 条可发布的短视频内容。

## 一眼看懂

`周一 选题` → `周二 脚本生成` → `周三 内容定稿` → `周四 视频生成` → `周五 发布` → `周六 观察` → `周日 周报`

## 周一｜提交选题

- 谁来做：内容负责人主导，大师/顾问补充。
- 做什么：提交本周 2 到 3 个核心概念，说明经典来源、现代角度、目标情绪。
- 在哪里做：优先用 GitHub Issue；临时灵感可先发微信，再补录到 Issue。
- 交接点：内容负责人确认信息完整后，创始人据此安排脚本生成。

## 周二｜跑脚本与初审

- 谁来做：创始人先跑管线，内容负责人开始初审。
- 做什么：根据选题生成脚本，检查内容是否贴近平台语境、是否清楚易懂。
- 在哪里做：脚本在内容管线和 GitHub 仓库中查看；反馈写在 GitHub Issue。
- 交接点：内容负责人把“可传播性”问题先标出来，交给创始人和大师/顾问继续处理。

## 周三｜文化复核与定稿

- 谁来做：大师/顾问主审，内容负责人汇总。
- 做什么：检查传统文化表达是否准确，避免过度承诺、迷信化或海外误读。
- 在哪里做：GitHub Issue 留下逐条意见；紧急澄清可走微信。
- 交接点：内容负责人汇总成最终修改意见，创始人据此定稿。

## 周四｜生成视频

- 谁来做：创始人主导，内容负责人确认交付细节。
- 做什么：把定稿脚本送入视频生成流程，检查视频状态、标题文案和排期。
- 在哪里做：内容管线、HeyGen、发布队列；必要反馈仍留在 GitHub Issue。
- 交接点：创始人确认视频可发布后，进入自动发布阶段。

## 周五｜开始发布

- 谁来做：系统自动发布，内容负责人跟进。
- 做什么：确认第一批内容按计划上线，查看是否有平台异常或文案问题。
- 在哪里做：社交平台后台、GitHub Issue、微信。
- 交接点：如发现异常，内容负责人先记录，创始人决定是否调整队列。

## 周六｜观察反馈

- 谁来做：内容负责人主导。
- 做什么：观察播放、点赞、评论、分享等数据，记录观众高频反馈。
- 在哪里做：TikTok / Instagram 后台，必要时把重点反馈记入 GitHub Issue。
- 交接点：把高价值反馈沉淀成下周选题线索。

## 周日｜提交周报

- 谁来做：内容负责人主导，创始人补充策略判断。
- 做什么：填写本周发布数量、最佳表现内容、观众反馈、下周方向与堵点。
- 在哪里做：GitHub Issue 的 Weekly Report 模板。
- 交接点：周报提交后，直接成为下周一选题准备材料。

## 三种沟通渠道怎么分

- GitHub Issue：用于可追踪的任务、反馈、周报和选题。
- 微信：用于紧急沟通、临时提醒和不能等的事情。
- 内容管线：由创始人操作，用于脚本生成、视频生成、发布与部署。

## 常见卡点

- 选题只有一句话，没有现代场景，脚本会变得空泛。
- 审核只说“感觉不对”，没有具体修改点，创始人无法高效处理。
- 文化问题只在微信口头提，没有同步到 Issue，后续无法追踪。
