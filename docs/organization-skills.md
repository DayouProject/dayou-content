# 组织级 AI Skills

这页定义的是 Dayou 希望所有 AI 客户端都共享的一套行为。

Dayou 当前的最低标准是：每个 collaborator 至少配置下面三者之一：

- Claude
- Cursor
- Codex

我们都希望它们至少会做同一批事，而不是每个工具都重新发明一套流程。

## 什么叫“组织级 skill”

在 Dayou 里，它不是指某个软件专属功能，而是指一套稳定行为：

- 什么时候该先判断 repo
- 什么时候该先整理问题
- 什么时候该转成 office hours 问题
- 什么时候该直接产出 GitHub Issue
- 什么时候该生成 `/daily`

## Skill 1：`/route`

### 目的

判断一个问题、反馈或机会应该进入哪个仓库。

### 输入

- 用户当前问题
- 影响对象
- 当前现状
- 想要的结果

### 输出

- 推荐仓库
- 判断理由
- 还缺什么信息
- 下一步动作

### 通过标准

看完输出后，用户应该知道自己下一步该去哪一个仓库，而不是继续悬空。

## Skill 2：`/context`

### 目的

把 `README / TODO / open Issues` 解释成普通中文，让非技术协作者也能读懂。

### 输入

- 仓库名称
- README、TODO、Issue 列表或链接

### 输出

- 当前仓库在推进什么
- 哪些事情已经在做
- 哪些事情还没开始
- 用户现在最值得关注的 3 个点

### 通过标准

用户不需要看代码，也能知道项目当前状态。

## Skill 3：`/issue`

### 目的

把模糊想法压缩成一个可以提交到 GitHub 的正式 Issue 草稿。

### 输入

- 当前问题或机会
- 为什么值得做
- 证据、截图、链接、例子
- done 标准

### 输出

- Issue 标题
- 现状
- 目标结果
- 优先级理由
- 证据
- done 标准

### 通过标准

用户复制后，只需要少量修改，就能正式提交到 GitHub。

## Skill 4：`/office-hours`

### 目的

把一个“值得问技术专家的问题”整理成高质量的 gstack office hours brief。

### 输入

- 当前需求
- 现有技术栈
- 不确定点
- 最小版本设想

### 输出

- 背景摘要
- 现有系统假设
- 最小可交付版本
- 需要确认的关键问题
- 希望带回 Dayou 的决策点

### 通过标准

用户拿着这份 brief 去问，不会浪费时间在重新解释背景上。

## Skill 5：`/daily`

### 目的

把当前上下文压缩成一个日更节奏，让 collaborator 知道今天该推进什么。

### 输入

- 当前仓库
- 当前 open Issues / TODO
- 昨天做了什么
- 当前 blockers

### 输出

- 今天关注的仓库
- 昨天
- 今天
- blockers
- 需要谁确认

### 通过标准

用户能直接把输出拿去做 daily，对齐团队节奏。

## Skill 6：`/next`

### 目的

当用户不知道下一步做什么时，给出最小可执行的三步。

### 输入

- 当前问题
- 当前卡点
- 当前仓库或候选仓库

### 输出

- 现在先做什么
- 做完后看什么结果
- 如果卡住，转向哪里

### 通过标准

用户能立刻开始动作，而不是继续停留在抽象讨论。

## Dayou 希望所有 AI 共享的 guardrails

每个 AI，不管是网页版还是代码工具，都应该默认遵守下面这些边界：

1. 默认用户可能不会代码
2. 默认用户不需要本地开发环境
3. 默认先用普通中文解释
4. 默认先看上下文，再给建议
5. 默认正式结论回到 GitHub Issue
6. 默认不要假定固定岗位
7. 默认不要让聊天替代留痕

## 四个统一触发词

为了让不同 AI 客户端行为尽量一致，Dayou 可以统一用下面四个触发词：

| 触发词 | AI 应该做什么 |
|---|---|
| `/route` | 判断仓库归属 |
| `/issue` | 产出 GitHub Issue 草稿 |
| `/office-hours` | 整理 gstack office hours brief |
| `/daily` | 生成今日 daily |

## 推荐的统一开场白

如果你想让不同 AI 尽量进入同一套 Dayou 模式，可以先发这段话：

```text
你现在进入 Dayou 协作模式。
默认用户不一定会代码。
请优先使用普通中文。
请先判断我需要的是 /route、/issue、/office-hours 还是 /daily。
所有正式结论最后都要回到 GitHub Issue。
```

## 这套 skills 的真正作用

它解决的不是“AI 会不会说话”，而是：

- 不同 AI 会不会把用户带到同一个流程里
- 不同 AI 会不会理解同一套仓库边界
- 不同 AI 会不会把结果收束到同一种正式留痕方式

如果这三件事统一了，客户端不同就不再是大问题。
