# 仓库分工 / Repo Scope

这页只回答一件事：三个核心仓库分别负责什么，什么应该放进去，什么不应该放进去。

## 总原则

- 三个仓库同等重要
- 它们不是主次关系，而是分工关系
- 提需求前，先判断它属于哪个仓库
- 不确定时，先问 AI，再开最相关的 Issue

## 一张表看懂

| 仓库 | 核心 scope | 应该放什么 | 不应该放什么 |
|---|---|---|---|
| `mysterious` | 主产品 | 排盘、AI 解读、用户体验、登录、支付、数据、自动化、主产品增长 | 纯宠物产品需求；团队手册本身 |
| `animal-dayou` | 宠物产品 | 宠物排盘、宠物用户体验、宠物场景文案、宠物产品增长 | 主产品通用功能；团队协作文档 |
| `dayou-content` | 协作中台与内容资产 | 团队手册、协作工作流、内容资产、知识库文档、入门说明、统一入口 | 主产品代码实现；宠物产品代码实现 |

## `mysterious` 的核心工作范围

适合放进 `mysterious` 的事情：

- 主产品功能
- 主产品页面和交互
- AI 解读质量
- 登录、支付、用户数据
- 与主产品直接相关的自动化
- 影响 `dayou.dev` 正式线上体验的需求

典型例子：

- “注册流程太绕，想减少流失”
- “AI 回复结构不稳定，需要更可控”
- “X 上来的用户落地页不够清楚”
- “想给主产品加一个更清晰的 onboarding”

不适合放这里的：

- 只是团队协作方式的问题
- 只是文档入口的问题
- 只和宠物产品有关的问题

## `animal-dayou` 的核心工作范围

适合放进 `animal-dayou` 的事情：

- 宠物产品相关功能
- 宠物用户的使用路径
- 宠物产品的增长、文案、体验
- 宠物产品的反馈与异常

典型例子：

- “宠物信息输入流程太复杂”
- “宠物产品的结果页解释不够清楚”
- “宠物方向 landing page 转化太低”

不适合放这里的：

- 主产品 `mysterious` 的功能或增长问题
- 团队手册和统一协作流程

## `dayou-content` 的核心工作范围

适合放进 `dayou-content` 的事情：

- 三个仓库共用的协作规则
- GitHub Pages 团队手册
- 零基础协作者 onboarding
- 内容资产和知识库文档
- 统一工作流说明
- “这个需求应该去哪一个仓库提” 这类元问题

典型例子：

- “新人如何加入协作体系”
- “不会写代码的人如何提需求”
- “三个仓库的边界怎么定义”
- “团队现在统一用什么流程推进需求”

不适合放这里的：

- 某个具体产品里的代码功能实现
- 只影响单个产品体验的功能需求

## 决策规则

如果一个需求同时影响多个仓库，用这个顺序判断：

1. 它最终改变的是哪个产品或系统？
2. 它的实现主要落在哪个仓库？
3. 哪个仓库的维护者需要长期追踪这件事？

如果答案仍然不清楚：

- 先在 `dayou-content` 把问题写清楚
- 或先问 AI，要求它帮你判断归属

## 一句话判断法

- 改主产品：去 `mysterious`
- 改宠物产品：去 `animal-dayou`
- 改协作方式、文档入口、内容资产：去 `dayou-content`
