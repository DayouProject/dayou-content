# gstack Office Hours

这页是 Dayou 的 `gstack office hours` 专用提问模板页。

目标不是“想到什么就去问什么”，而是先把问题压成一个高质量 brief，再去问。

## 什么时候该用

适合这些情况：

- 需求方向合理，但技术路径不确定
- 想做自动化，但不知道现有技术栈是否合适
- 想确认第一版最小可交付应该切到哪里
- 想确认应该继续堆现有系统，还是另起一层轻量服务

不适合这些情况：

- 你还没判断事情属于哪个仓库
- 你只是还没把问题讲清楚
- 你还没看过 README、TODO、open Issues
- 你其实已经可以直接开 GitHub Issue

## 先用 AI 整理，再去问

默认顺序：

1. 先让 AI 帮你做 `repo 路由`
2. 再让 AI 把问题整理成 office hours brief
3. 再去问 gstack
4. 问完以后，把结论带回 GitHub Issue

## 你要带去 office hours 的，不是情绪，是这 6 件事

| 项目 | 你至少要说清楚什么 |
|---|---|
| 背景 | 现在团队在推进什么，为什么会出现这个问题 |
| 当前问题 | 哪一步慢、乱、容易丢信息，或者不可扩展 |
| 现有技术栈 | 现在已经在用什么，不想重复造什么 |
| 最小版本假设 | 第一版到底只想做到哪里 |
| 约束 | 时间、权限、平台、维护成本、组织方式 |
| 希望确认 | 你真正想从 office hours 带回来的 3 到 5 个答案 |

## 复制即用的 AI 指令

先把这段话发给你的 AI：

```text
请把下面这个问题整理成 Dayou 可直接拿去问 gstack office hours 的 brief。

要求：
1. 用普通中文
2. 先帮我补齐背景、现有技术栈、最小版本、约束
3. 最后只保留最值得问的 3 到 5 个问题
4. 不要把问题写得太虚
5. 输出格式必须是：

背景：
当前问题：
现有技术栈：
最小版本假设：
约束：
最想确认的问题：
会后要带回 GitHub 的结论：
```

## 模拟用例

下面这个例子是**模拟用例**，用于展示提问质量，不代表当前线上真实状态。

### 原始想法

```text
我们很多需求先在聊天里讨论，再手工整理成 Issue，容易丢上下文。
我想知道能不能做一个轻量 intake pipeline：
让 collaborator 只需要说自然语言，
AI 先判断应该进哪个仓库，再生成 Issue 草稿，
必要时再给出 office hours 问题。

我不确定这件事应该继续堆在现有 GitHub Pages + GitHub Issue 流程上，
还是需要额外做一层服务。
```

### 不够好的问法

```text
我们想做自动化，这个能做吗？
```

问题在于：

- 没有背景
- 没说现有系统
- 没说最小版本
- 没说你真正想确认什么

### 整理后的 office hours brief

```text
背景：
Dayou 现在有三个平级仓库，协作者默认通过自然语言 + AI + GitHub Issue 参与推进。
当前最大摩擦点是：模糊需求先在聊天里被讨论，之后还要手工整理成正式 Issue，容易丢上下文，也容易提错仓库。

当前问题：
我们想降低非技术 collaborator 的进入门槛，让他们只需要通过自然语言描述问题，
AI 就能先做 repo 路由、生成 Issue 草稿，并在技术路径不清楚时自动切到 office hours 准备。
不确定这应该继续依附在现有文档与 GitHub 流程上，还是应该额外做一层 intake service。

现有技术栈：
GitHub 仓库 + GitHub Pages 文档站 + GitHub Issues 作为正式留痕。
协作者默认通过 ChatGPT / Claude / Cursor 等 AI 工具参与。
产品仓库与部署能力已存在，但我们不希望第一版把本地开发门槛推给 collaborator。

最小版本假设：
第一版只做三件事：
1. 读取 Dayou 的协作规则
2. 帮用户判断该进哪个仓库
3. 生成可复制到 GitHub 的 Issue 草稿
如果技术可行性不确定，再输出 office hours brief，而不是自动创建复杂系统。

约束：
1. 默认用户不会代码
2. 不希望要求本地开发环境
3. 正式结论仍然必须回到 GitHub Issue
4. 第一版希望维护成本尽量低

最想确认的问题：
1. 这个问题更适合先做“prompt + 文档 + GitHub Issue 模板”层，还是值得直接做单独 intake service？
2. 如果先做最小版本，最合理的边界应该切在哪里，避免过早工程化？
3. 哪些能力必须自动化，哪些能力应该继续留在人和 AI 的对话层？
4. 如果未来要扩展成多 agent 协作，这一层最该先标准化什么？

会后要带回 GitHub 的结论：
- 值不值得做
- 第一版边界
- 推荐路径
- 暂时不要做的部分
```

## 一个好的 office hours 结果，回来后应该怎么落地

不要只留在聊天记录里。

回来后至少做这两件事：

1. 把结论压缩成 GitHub Issue 评论或新 Issue
2. 明确写出：
   - 推荐路径
   - 第一版不做什么
   - 下一步谁来推进
